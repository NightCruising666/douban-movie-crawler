# 全流程筛选与方案决策记录

本文记录每个阶段筛选了什么、采用什么标准、为什么这样做，以及比较过哪些替代方案。它用于后续实验报告的方法章节。

## 阶段一：候选电影池

### 实际做法

- 从豆瓣电影搜索接口采集19个标签；
- 每个标签分页获取，保留标签内排名和采集时间；
- 以豆瓣ID去重后得到2601部独立候选，共3795条“电影—标签”关联；
- `movies_raw.csv` 保存候选实体，`movie_tags.csv` 保存来源和排名。

### 选择原因

- 多标签比只抓“热门”更能覆盖经典、高分、地区和类型差异；
- 豆瓣ID是稳定主键，可避免同名电影误合并；
- 标签来源和排名能用于评估样本偏差。

### 比较过但未采用的方法

- **只抓热门榜**：实现简单，但会过度集中于大众作品；
- **按时间参数建立独立样本层**：小规模验证发现与热门结果高度重合，不能假装成独立抽样；
- **按电影名称去重**：会误删同名不同年份作品；
- **把样本解释为完整市场**：标签搜索不是随机抽样，因此只研究“豆瓣标签样本”。

## 阶段二：电影详情

### 实际做法

- 对阶段一全部2601个ID采集15个原始字段；
- 不再按相关性截取前500部，最终覆盖全部候选；
- 以 `movies.csv` 中已有ID作为断点，边爬边存；
- 请求失败进行随机长冷却和重试；
- HTTP 400/404/410只有连续两个不同轮次失败才可判定为不可用；
- 最终2601部全部成功，12部曾失败后恢复，不可用0部。

### 选择原因

- 早期前500部相关性样本偏向近两年，无法支撑2005—2025趋势分析；
- 全量采集减少人为筛选，并保留老片、小众片；
- 两轮确认避免把临时限流或网络故障误判为数据不存在；
- 失败事实表与逐次尝试表为报告提供反爬处理证据。

### 比较过但未采用的方法

- **只采相关性前500部**：时间结构严重偏向近期，已放弃；
- **第一次HTTP 400就删除**：可能把临时接口异常当永久失败；
- **请求失败后立即高速重试**：容易加重限流；
- **代理IP自动轮换**：合规和安全风险高，本项目不采用。

## 阶段三：热门短评样本（尚未开始正式新版采集）

### 当前设计

- 每部电影定额采集热门排序前30条短评；
- 保存短评ID、电影ID、星级、正文、有用数、时间、排序位置和采集时间；
- 不保存用户名、用户ID、头像或其哈希；
- 评论样本只辅助课题A，不替代电影总体评分与评价人数。

### 选择原因

- 定额采样便于控制请求量和电影间可比性；
- 星级与有用数可以比较普通平均和有用数加权平均；
- 电影详情中的短评总数、长评总数反映总体互动规模，样本文本反映热门评论层。

### 比较过但未采用的方法

- **采完所有短评和长评**：请求量过大，超出课程项目边界；
- **把热门前30条当全体评论**：热门排序存在互动偏差，只能称“热门短评样本”；
- **原始有用数直接作为唯一权重**：极端高有用数会支配结果，因此同时计算普通、原始权重和对数权重；
- **保存用户标识**：与研究目标无关，增加隐私风险。

## 数据清洗阶段

### 实际做法

- 原始表不修改，清洗结果写入 `data/processed/`；
- 不按统一规则粗暴删行，而是生成质量标记；
- 课题A和课题B分别使用独立纳入标志；
- 清洗后保留2601部，课题A有效评分样本2590部，课题B时间范围样本1828部。

### 选择原因与替代方案

具体规则、实际命中数和未采用方法见 [CLEANING_PLAN.md](CLEANING_PLAN.md) 与运行后生成的 `movies_cleaning_rules.csv`。

## 后续统计与报告边界

- 相关系数描述关联，不解释因果；
- 贝叶斯加权用于稳定小样本评分，不直接判定“刷分”；
- 类型占比是当前豆瓣样本内占比，不代表完整电影市场份额；
- 短评有用数表示社区互动，不等价于评论真实性；
- 所有数量均标注采集时点，因为评分和评论总数会继续变化。
